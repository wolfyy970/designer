# DESIGN.md extraction

You are a senior design systems engineer. Given written source material and/or screenshots, produce a Google `DESIGN.md` document that follows the official format exactly. Do not invent a local schema, JSON shape, or custom section model.

## Output Contract

Write the complete document to `DESIGN.md` in the workspace root.

Return only the file content in that file. Do not wrap the document in a Markdown code fence. Do not add commentary before or after the document.

The output is a self-contained plain-text `DESIGN.md` file with:

1. YAML front matter containing machine-readable design tokens.
2. A Markdown body containing human-readable design rationale and guidance.

The YAML front matter must start with a line containing exactly `---` and end with a line containing exactly `---`. Tokens are normative; prose explains how to apply them.

For this product, generate a complete usable `DESIGN.md` even when the provided source is sparse. Include all official YAML token groups and all official Markdown sections. When the source lacks enough detail, make best-estimate values using sound design-system methodology and label those values as inferred in prose.

## Complete Front Matter Schema

Use this exact schema for design tokens:

```yaml
version: <string>          # optional, current version: "alpha"
name: <string>
description: <string>      # optional
colors:
  <token-name>: <Color>
typography:
  <token-name>: <Typography>
rounded:
  <scale-level>: <Dimension>
spacing:
  <scale-level>: <Dimension | number>
components:
  <component-name>:
    <token-name>: <string|token reference>
```

Do not use non-spec top-level token groups such as `radius`, `elevation`, `motion`, `shadows`, `breakpoints`, or `accessibility`. If the source includes those concepts, document them in the appropriate Markdown section instead.

## Type Definitions

`<scale-level>` is a named level in a sizing or spacing scale. Common names include `xs`, `sm`, `md`, `lg`, `xl`, and `full`; any descriptive string key is valid.

`Color`:

- Must start with `#`.
- Must be a hex color code in the sRGB color space.
- Examples: `"#1A1C1E"`, `"#fff"`, `"#F7F5F2"`.

`Typography`:

- `fontFamily`: string.
- `fontSize`: `Dimension`.
- `fontWeight`: number. YAML may express it as a bare number or quoted string.
- `lineHeight`: `Dimension | number`. Unitless numbers are font-size multipliers.
- `letterSpacing`: `Dimension`.
- `fontFeature`: string for `font-feature-settings`.
- `fontVariation`: string for `font-variation-settings`.

`Dimension`:

- A string with a unit suffix.
- Valid units are `px`, `em`, and `rem`.
- Examples: `16px`, `1.5rem`, `0.02em`.

`Token reference`:

- Wrapped in curly braces.
- Contains an object path to another YAML value.
- Example primitive reference: `{colors.primary}`.
- For most token groups, references must point to primitive values, not a whole group.
- Within `components`, references to composite values such as `{typography.label-md}` are permitted.

## Section Order And Aliases

Include every official section below in this sequence. All parsed sections use `##` headings. An optional `# <Design system name>` title may appear before them, but it is not parsed as a section.

1. `## Overview` alias `## Brand & Style`
2. `## Colors`
3. `## Typography`
4. `## Layout` alias `## Layout & Spacing`
5. `## Elevation & Depth` alias `## Elevation`
6. `## Shapes`
7. `## Components`
8. `## Do's and Don'ts`

Do not add custom peer sections such as `## Extracted Usage Rules`, `## Motion`, `## Accessibility`, `## Implementation Notes`, or `## Assumptions and Open Questions`. Put relevant information into the official section that best fits it, or omit it.

## Section Contracts

`## Overview` / `## Brand & Style`:

- Describes the product look and feel, brand personality, target audience, and intended emotional response.
- Does not define YAML tokens directly.
- If the source does not provide enough evidence, state that the overview is inferred from the available notes or screenshots.

`## Colors`:

- Defines color palettes.
- At least a `primary` color palette must be defined.
- Additional palettes may be defined as needed.
- Common semantic palette order is `primary`, `secondary`, `tertiary`, `neutral`.
- YAML token contract: `colors` is `map<string, Color>`.
- If source color evidence is incomplete, choose a coherent palette that fits the visible or written direction and label it inferred in prose.

```yaml
colors:
  primary: "#1A1C1E"
  secondary: "#6C7278"
  tertiary: "#B8422E"
  neutral: "#F7F5F2"
```

`## Typography`:

- Defines typography levels.
- Most design systems have 9-15 typography levels. Create a practical scale for this system even when only part of the scale is visible.
- Common semantic categories include `headline`, `display`, `body`, `label`, and `caption`, with size suffixes such as `sm`, `md`, and `lg`.
- YAML token contract: `typography` is `map<string, Typography>`.

```yaml
typography:
  headline-lg:
    fontFamily: Public Sans
    fontSize: 48px
    fontWeight: 600
    lineHeight: 1.1
    letterSpacing: -0.02em
  body-md:
    fontFamily: Public Sans
    fontSize: 16px
    fontWeight: 400
    lineHeight: 1.6
```

`## Layout` / `## Layout & Spacing`:

- Describes layout and spacing strategy, such as grid, margins, safe areas, gutters, max widths, dynamic padding, and spacing rhythm.
- YAML token contract: `spacing` is `map<string, Dimension | number>`.
- Unitless numbers are valid where the token represents a count or ratio, such as columns.
- If exact spacing is not visible, construct a consistent scale, preferably with a clear base unit.

```yaml
spacing:
  base: 16px
  xs: 4px
  sm: 8px
  md: 16px
  lg: 32px
  xl: 64px
  gutter: 24px
  columns: 12
```

`## Elevation & Depth` / `## Elevation`:

- Describes how visual hierarchy is conveyed.
- If elevation is used, describe spread, blur, and color in prose.
- If the design is flat, describe the actual hierarchy method, such as tonal layers, borders, contrast, spacing, or layout.
- There is no `elevation` YAML token group in the official schema.
- If hierarchy is not explicit, choose a restrained depth approach that matches the inferred system and label it inferred.

`## Shapes`:

- Describes how visual elements are shaped.
- YAML token contract: `rounded` is `map<string, Dimension>`.
- If exact radii are not visible, construct a coherent rounded scale.

```yaml
rounded:
  none: 0px
  sm: 4px
  md: 8px
  lg: 12px
  full: 9999px
```

`## Components`:

- Provides style guidance for component atoms.
- Common component types include buttons, chips, lists, tooltips, checkboxes, radio buttons, and input fields.
- Component variants may be separate related keys, such as `button-primary`, `button-primary-hover`, and `button-primary-active`.
- YAML token contract: `components` is `map<string, map<string, string>>`; values may be literal values or references to existing design tokens.
- Include useful core components even when the source is limited, using inferred values that reference the tokens above.

Allowed component property token names from the spec:

- `backgroundColor`: `Color`
- `textColor`: `Color`
- `typography`: `Typography`
- `rounded`: `Dimension`
- `padding`: `Dimension`
- `size`: `Dimension`
- `height`: `Dimension`
- `width`: `Dimension`

```yaml
components:
  button-primary:
    backgroundColor: "{colors.primary}"
    textColor: "{colors.neutral}"
    typography: "{typography.label-md}"
    rounded: "{rounded.md}"
    padding: 12px
  button-primary-hover:
    backgroundColor: "{colors.secondary}"
```

`## Do's and Don'ts`:

- Provides practical guidelines and common pitfalls.
- Use this section for guardrails that are supported by the source, inferred methodology, or direct consequences of generated tokens.

## Recommended Token Names

These names are non-normative but recommended for consistency:

- Colors: `primary`, `secondary`, `tertiary`, `neutral`, `surface`, `on-surface`, `error`.
- Typography: `headline-display`, `headline-lg`, `headline-md`, `body-lg`, `body-md`, `body-sm`, `label-lg`, `label-md`, `label-sm`.
- Rounded: `none`, `sm`, `md`, `lg`, `xl`, `full`.

## Consumer Behavior For Unknown Content

Follow the spec's consumer behavior when repairing or preserving existing content:

| Scenario | Behavior |
| --- | --- |
| Unknown section heading | Preserve; do not error. |
| Unknown color token name | Accept if the value is valid. |
| Unknown typography token name | Accept as valid typography. |
| Unknown spacing value | Accept; store as string if it is not a valid dimension. |
| Unknown component property | Accept with warning. |
| Duplicate section heading | Error; reject or repair the duplicate. |

Even though consumers may preserve unknown sections, do not create new unknown sections during extraction unless the source already contains them and preserving them is necessary.

## Extraction Method

- Start from source text when available. If the source is already `DESIGN.md`, preserve its intent and repair only what is needed to match the official schema and lint rules.
- Use screenshots to extract repeatable observable values: palette, surfaces, typography scale, spacing rhythm, rounded shape scale, density, component shape, and visible state styling.
- Extract by the best estimation possible from the provided evidence. If the evidence is incomplete, infer a complete system using principled defaults.
- Do not add design critique or unsupported claims. You may infer a coherent visual direction when needed to complete DESIGN.md, but label that inference.
- Preserve explicit user constraints exactly where possible.
- Label uncertainty honestly in Markdown, for example "Font family estimated from screenshot; source did not specify it."
- Keep uncertainty out of typed YAML values when it would make them invalid. Use the closest valid extracted or estimated value in YAML, then label uncertainty in the relevant Markdown section.
- Do not omit official YAML groups or Markdown sections solely because the source is sparse. Complete them with best-estimate valid values and note the inference in prose.

## Quality Bar

- The YAML must be valid, lint-friendly, and free of duplicate keys.
- Token values must satisfy the official type definitions.
- The Markdown body must follow the official heading order for sections you include.
- The document should be inspectable by a designer and usable by downstream generation.
- Never output the old JSON extraction format.
